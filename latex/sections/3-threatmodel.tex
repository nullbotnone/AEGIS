\section{Threat Model}

\subsection{The Hijacked Agent Threat}

\begin{figure}[!t]
    \centering
    \includegraphics[width=0.95\columnwidth]{figures/threat_model.pdf}
    \caption{The hijacked authorized agent threat. An agent with valid credentials is subverted through injection attacks. The hijacked agent exfiltrates data through encrypted LLM API channels invisible to traditional data loss prevention and monitoring systems.}
    \label{fig:threats}
\end{figure}

We identify the most dangerous threat to HPC environments deploying AI agents: \textit{the hijacked authorized agent}, as illustrated in Figure~\ref{fig:threats}. Unlike malicious agents, which are blocked by existing authentication and authorization mechanisms, a hijacked agent operates under the full credentials and privileges of a legitimate user. It is not an intruder; it is a trusted insider that has been turned.

This threat arises from prompt injection~\cite{he2025security, greshake2023not} and tool poisoning~\cite{xie2025security} attacks exploiting known vulnerabilities in agent architectures~\cite{wu2025assistants} against LLM-based agents. An attacker crafts inputs, through data files, tool outputs, shared documents, or collaborative channels, that subvert the agent's instruction-following behavior. The agent, now under adversarial control, executes commands indistinguishable from the legitimate user's intent.

A hijacked agent possesses four properties that make it uniquely dangerous in HPC. \circled{1} \textbf{Full credential inheritance}: the agent operates with the user's Kerberos tickets, SSH keys, and scheduler permissions. No privilege escalation is needed; the agent already has access. From the access control system's perspective, every action is authorized. \circled{2} \textbf{Cross-project filesystem access}: HPC shared filesystems are typically organized by user, not by project. A user authorized on Projects A, B, and C grants their agent simultaneous access to all three datasets. A hijacked agent can traverse project boundaries that would require separate authorization in a properly segmented system. \circled{3} \textbf{Appearance of legitimacy}: the agent runs under an authorized user identity, invokes authorized tools, and follows authorized workflow patterns. Traditional anomaly detection fails because the agent's observable behavior including job submission, file I/O, and network access is consistent with its authorized role. \circled{4} \textbf{Exfiltration through the LLM API channel}: the agent communicates with its LLM backend via HTTPS API calls. Sensitive data can be encoded into prompts or tool outputs and transmitted through this channel, which is encrypted (invisible to network-level DLP), whitelisted (agents must communicate with the LLM to function), high-bandwidth (prompts can contain large context windows), and attributable to normal operation (no anomalous network destination).

This exfiltration vector is invisible to traditional Data Loss Prevention (DLP) systems, which inspect network traffic for sensitive data patterns. The LLM API channel bypasses this by design.

\subsection{Threat Scenarios}

We describe four concrete attack scenarios that illustrate the hijacked authorized agent threat in HPC environments.

In a data file injection scenario, an attacker embeds a prompt injection payload in a scientific dataset such as a comment field in a FITS header or a markdown cell in a Jupyter notebook. When the agent processes this file as part of a workflow, the injection hijacks the agent's subsequent actions. The agent treats the data as authoritative scientific input and executes the embedded commands.

In a tool output poisoning scenario, a compromised or adversarial tool returns output containing hidden instructions. The agent, treating the tool output as trusted data, executes the injected commands. This differs from data file injection in that the attack vector originates from code execution rather than data processing.

In a collaborative channel attack scenario, common in multi-agent workflows, a compromised agent in one project injects instructions through inter-agent communication channels. The hijack propagates laterally across project boundaries, allowing an attacker to escalate from a single compromised agent to access across multiple projects.

In a supply chain compromise scenario, a malicious update to an agent framework, dependency library, or model weights introduces a backdoor that activates when specific conditions are met. The backdoor exfiltrates data through the normal API channel, making the attack appear as legitimate agent behavior.

\subsection{Adversarial Capabilities}

We consider an adversary with several practical capabilities. They can craft inputs that manipulate an agent’s instruction-following behavior through prompt injection. With network access, they may observe metadata such as traffic timing and volume, though not the encrypted contents themselves. The adversary does not have direct access to an HPC account and must act through compromised or hijacked agents. They may also maintain persistence across multiple agent sessions and job submissions.

At the same time, we assume certain boundaries. The adversary cannot compromise the scheduler or resource manager, does not have access to the LLM provider’s infrastructure, and cannot break hardware-based roots of trust such as TPMs or secure enclaves.

\subsection{Why Existing Defenses Fail}

Authentication and authorization both succeed because the agent is operating with valid credentials and legitimate permissions. However, this is precisely what allows the attack to remain undetected. Network monitoring is ineffective, as LLM API traffic is both encrypted and explicitly whitelisted. Similarly, DLP mechanisms fail because sensitive data can be embedded within these encrypted API calls. User behavior analytics also struggle to identify anomalies, since the agent’s actions remain consistent with expected workflow patterns. Even sandboxing provides limited protection, as the agent must retain access to the filesystem and network to perform its intended tasks. Taken together, these limitations highlight the need for \textbf{attestation: continuous verification that the agent’s behavior conforms to its authorized intent, not just that its identity is valid}.

\subsection{Unique Properties of HPC Agent Injection Attacks}

\begin{figure}[!t]
    \centering
    \includegraphics[width=0.90\columnwidth]{figures/injection_surfaces.png}
    \caption{Four HPC-specific injection attack surfaces. (1) Filesystem-mediated: poisoned data in shared parallel filesystems. (2) Co-location: shared /tmp on multi-tenant compute nodes. (3) Supply chain: compromised agent tools. (4) Coordinated: multi-agent covert exfiltration network.}
    \label{fig:injections}
\end{figure}

Agent injection attacks in HPC exploit \textit{shared infrastructure} in ways not studied in prior work on prompt injection, which focuses on web-based and chatbot scenarios. These properties arise from the unique characteristics of HPC infrastructure: shared filesystems, multi-tenant compute nodes, and emerging agent skill ecosystems, as illustrated in Figure~\ref{fig:injections}.

\circled{1} \textbf{Filesystem-mediated injection} arises from a core property of HPC systems: shared storage. Filesystems such as Lustre introduce injection surfaces that have no direct counterpart in web environments. An attacker with access to a shared project space can plant adversarial content (e.g., poisoned metadata, hidden instructions embedded in log outputs) that a target agent later reads as if it were legitimate scientific data. Because HPC workflows depend on treating shared data as trustworthy, this assumption cannot be easily relaxed without disrupting normal operation.

\circled{2} \textbf{Multi-user co-location injection} stems from how schedulers pack jobs onto shared nodes. Jobs from different users often run side by side, enabling application-level side channels. An attacker can write adversarial content to shared locations such as \textit{/tmp}, which a co-located agent may subsequently read. This does not require access to the target’s project data, only that both jobs land on the same node.

\circled{3} \textbf{Supply chain injection via agent skills} reflects risks introduced by tool ecosystems such as OpenClaw, LangChain, and MCP servers. These components resemble traditional software supply chains: if a tool is compromised, it can inject malicious instructions directly into the agent’s control loop via its outputs. Since the agent intentionally invoked the tool, the resulting behavior appears indistinguishable from normal execution.

\circled{4} \textbf{Coordinated multi-agent exfiltration} captures attacks that span multiple agents. Compromised agents across different users or projects can cooperate to move data covertly. One agent may stage sensitive data in a shared location, while another retrieves it and exfiltrates it through its own LLM API channel. Each agent’s behavior, viewed in isolation, appears benign; the attack only becomes apparent when considering interactions across agents.
