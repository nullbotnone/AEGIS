\section{Background}

This section provides background on zero-trust security, AI agents and workflow automation, the security posture of contemporary HPC systems, and prompt injection in LLM-integrated applications.

\subsection{Zero-Trust Architecture}

Zero-trust architecture replaces coarse network perimeters with continuous policy decisions over identities, devices, services, and resources. The original zero-trust framing emphasized the idea that location on a trusted network should not be treated as a security boundary~\cite{kindervag2010zero}. NIST SP 800-207 later formalized zero trust as an architecture centered on explicit authentication, authorization, and continuous evaluation~\cite{rose2020zta}. NIST SP 800-207A extended this framing to cloud-native applications, highlighting workload identity, application telemetry, and fine-grained service-to-service enforcement~\cite{chandramouli2023zta}.

These ideas are increasingly relevant to HPC, but they do not directly solve the autonomous-agent problem. Existing zero-trust deployments focus on who is requesting access and under what context. In an AI-agent setting, that is necessary but insufficient: a legitimate, correctly authenticated agent may still behave outside its authorized task after being hijacked by adversarial inputs.

\subsection{AI Agents and Scientific Workflow Automation}

Classical agent research studies autonomous entities that perceive, reason, and act to achieve goals~\cite{wooldridge2009agents}. Modern LLM-based agents extend this model with tool use, planning, reflection, and long-horizon execution~\cite{schick2023toolformer,yao2023react,wang2023voyager,mialon2023augmented,xi2023agentsurvey,masterman2024agentsurvey}. These systems increasingly rely on external tools and protocol-mediated service access, which expands both their capability and their attack surface~\cite{mcp2024spec}.

HPC environments already depend on large-scale automation through workflow and scheduling systems such as Pegasus, Parsl, FireWorks, and \slurm{}~\cite{deelman2015pegasus,babuji2019parsl,jain2015fireworks,yoo2003slurm}. AI agents are a natural next step in that automation stack because they can adapt plans at runtime rather than executing only pre-declared DAGs. However, this same flexibility means that an attacker can steer them through data, tool outputs, or communication channels that the surrounding infrastructure still treats as legitimate workflow inputs.

\subsection{Security in HPC}

Contemporary HPC sites remain strongly identity- and scheduler-centric. Authentication often relies on Kerberos, SSH, and federated research identity infrastructure such as CILogon~\cite{neuman1994kerberos,basney2014cilogon}. Authorization is enforced through scheduler policy, POSIX permissions, ACLs, and shared storage semantics. Storage and execution environments are optimized for performance and collaboration rather than for strict tenant isolation; in practice, sites commonly rely on \slurm{}, Lustre, and IBM Spectrum Scale or related shared filesystems~\cite{yoo2003slurm,lustre2025intro,ibm2024spectrumscale}.

This model works reasonably well for human-driven batch computing, but it leaves an important gap for autonomous agents. Once a user launches an agent under valid credentials, the agent inherits the user's authority across files, tools, and network endpoints. The system verifies identity and resource ownership, but it does not continuously verify whether the resulting sequence of actions remains faithful to the user's task.

\subsection{Prompt Injection and Agent Security}

Prompt injection has emerged as a fundamental security problem for LLM-integrated applications because language models do not inherently distinguish trusted instructions from adversarial content embedded in data~\cite{greshake2023indirect,liu2023houyi,liu2024prompt}. The broader risk picture for LLM systems includes misuse, prompt leakage, jailbreaks, and downstream tool abuse~\cite{perez2022ignore,weidinger2022taxonomy,owasp2025llm}. Universal adversarial suffixes further show that LLM guardrails are brittle even when developers try to enforce instruction hierarchies purely in the prompt~\cite{zou2023universal}.

A growing literature explores partial defenses and evaluation frameworks. Spotlighting improves source separation between trusted instructions and untrusted data~\cite{hines2024spotlighting}, while StruQ proposes structured prompting to separate instructions from data payloads~\cite{chen2024struq}. ToolEmu and AgentDojo provide controlled environments for surfacing safety failures and adversarial behavior in tool-using agents~\cite{ruan2023toolemu,debenedetti2024agentdojo}. These works establish the threat and improve evaluation practice, but they do not directly address the cluster-level runtime enforcement problem studied in this paper.
